SWE sub-agents — parallel by design

Backend, Frontend, and Mobile are sub-agents. The SWE Coordinator spawns only the relevant ones, runs them in parallel, then synthesises the debate.

Scope-based dispatch

Only spawn sub-agents whose domain is touched by the task. No wasted context, no irrelevant opinions.

# Web + backend feature
⚙️ Backend → spawned
🖥️ Frontend → spawned
📱 Mobile → skipped (no native surface)

# Mobile-only bug
📱 Mobile → spawned
⚙️ Backend → skipped (API unchanged)
🖥️ Frontend → skipped

# Full-stack feature
⚙️ Backend → spawned ↓ all three
🖥️ Frontend → spawned ↓ parallel
📱 Mobile → spawned ↓ debate

Three-way challenge loop

Sub-agents read real code first, then challenge each other. Better options emerge from the friction.

[Backend → Frontend]: your API usage requires
  3 round-trips to render one screen. I can add
  a composite endpoint but it'll break REST purity.

[Frontend → Backend]: agreed on composite endpoint.
  But also: your error shape doesn't include a
  retry-after header. That's a UX blocker.

[Mobile → Backend]: response is 340KB on this
  endpoint. Kills battery on 3G polling. Need a
  delta/patch API or push instead of poll.

[Backend → All]: proposing Option 2 — composite
  endpoint + delta endpoint + retry-after header.

Principal Engineer. Reviews all 4 engineering artifacts for cross-artifact consistency. Challenges each independently. Writes the final approved spec when all pass.
Engineering Gate

Two Bar Raisers

Adversarial reviewers. Their challenges are blockers, not suggestions. Max 2 rounds per artifact — then they escalate to you with a specific question.

Product Track

Product Bar Raiser

Principal PM / Head of Product. Gates the product track — engineering cannot start until PM and Designer are both approved.

  • Challenges every AC that is untestable or unmeasurable
  • Verifies all personas are covered: end user, ops/admin, support team
  • Checks every journey step has error states and recovery actions
  • Web-searches competitor references — verifies claims against real product behavior
  • Challenges API contracts for missing error cases or impossible data states
  • Verifies ops team can actually manage this feature after launch
Challenge → Producer revises → BR re-reviews · max 2 rounds · then escalates to you
Engineering Track

Engineering Bar Raiser

Principal Engineer. Reviews all four engineering artifacts together. Writes the final spec when all approved.

  • Challenges SWE options for vague tradeoffs or inconsistency with design
  • Challenges Staff SWE concerns that lack codebase evidence
  • Challenges SRE rollback plans that are theoretical rather than executable
  • Challenges QA for ACs without tests or TODOs instead of test code
  • Cross-checks all four artifacts for consistency with each other
  • SWE recommendation must match Staff SWE architecture — flags divergence
Challenge → Producer revises → BR re-reviews · max 2 rounds · then escalates to you

Register your repos

Engineer sub-agents read real code before proposing anything. Tell them where your repos live — once per machine.

1

Run the setup command

Prompts you for backend, web, and mobile repo paths. Validates each path exists.

2

Paths saved to .brocode-repos.json

File is gitignored — paths are machine-local. Re-run anytime to update.

3

Sub-agents read their repo automatically

Backend reads backend path. Frontend reads web. Mobile reads mobile. If a path isn't set, the agent asks before proceeding.

/brocode repos

Context files

Every run creates .sdlc/<id>/ — small focused files, one per agent. Gitignored. Resume anytime.

.sdlc/spec-20260426-oauth/
  00-tpm-log.md               ← TPM master log, live throughout
  00-brief.md                  ← user input + clarified scope
  00-input-raw.md              ← ingested doc / image
  01-requirements.md           ← PM, versioned
  02-design.md                 ← Designer, versioned
  03-implementation-options.md ← SWE team debate output
  04-architecture.md           ← Staff SWE, versioned
  05-ops.md                    ← SRE, versioned
  06-test-cases.md             ← QA, versioned
  threads/
    product-conversation.md     ← PM ↔ Designer
    swe-debate.md               ← Backend ↔ Frontend ↔ Mobile
    eng-conversation.md         ← SWE ↔ Staff SWE ↔ SRE ↔ QA
    eng-product-conversation.md ← Eng ↔ PM/Designer
  07-product-br-reviews/        ← challenge rounds + gate approval
  07-eng-br-reviews/            ← challenge rounds + approvals
  08-final-spec.md             ← approved, ready to implement

Input formats

Paste, attach, or link. brocode handles the rest. Never silently fails — if an MCP is missing, it tells you exactly what to do.

💬

Plain text

Describe the bug or feature in plain English — as much or as little detail as you have

🖼️

Image / Screenshot

Attach Figma exports, error screens, or mockups — PM and Designer analyze via vision

📄

Google Doc

Paste a link — fetched automatically via Google Drive MCP

Requires Google Drive MCP
📝

Notion / Wiki

No MCP by default — brocode asks you to paste the content directly

🎨

Figma

No Figma MCP by default — brocode asks you to export as PNG and attach
