Install

1
Clone and run the bootstrap script
git clone https://github.com/im-adarsh/brocode && cd brocode && bash install.sh

Copies all hooks/*.js to ~/.claude/hooks/. If ~/.claude/settings.json already exists it is left untouched — you get a diff command to review and merge manually.

2
Fill in your API keys in ~/.claude/settings.json
"ANTHROPIC_API_KEY": "sk-ant-..." ~/.claude/settings.json
3
Install GSD and any missing plugins inside Claude Code
/plugins install gsd@claude-plugins-official
4
Launch Claude Code — everything activates automatically
claude

Hooks

All bc-*.js hooks live in hooks/ in this repo and are installed to ~/.claude/hooks/ by install.sh. They form a layered enforcement pipeline that fires before and after Claude touches files.

Execution order. On every PreToolUse event the chain runs in sequence: bc-programmer → bc-planner → bc-secret-guard → bc-test-review → bc-commit-msg → bc-branch-guard → bc-lint. Any hook can hard-block with a message. bc-doc-sync fires on Stop.

bc-programmer PreToolUse

The primary evaluation gate. Fires once per session on the first Write / Edit / MultiEdit / NotebookEdit. Reads the conversation transcript, classifies task complexity and type, recommends the right workflow or skill, and requires Claude to answer five completeness questions before any file is touched. Also suppresses bc-planner by writing its flag file.

Task signal detected in transcriptRecommended workflow
"new project / app / system / from scratch / greenfield"/gsd:new-project
"new feature / refactor / migrate / redesign / rewrite"/gsd:plan-phase
"bug / debug / not working / broken / error / crash"/gsd:debug
"ui / component / design / css / layout / frontend"/frontend-design skill
"review / PR / audit / check code / pull request"/pr-review-toolkit skill
"implement / build / create / add / integrate / develop"/feature-dev skill
"monitor / watch / poll / every N / schedule / recurring"/loop skill
"update / change / rename / fix (small)"/gsd:quick
focused or trivial taskDirect (with written plan)

Completeness gate — Claude must write out answers to all five before proceeding:

  1. Are you using the recommended workflow (or have a clear reason to deviate)?
  2. Have you read every file relevant to this task?
  3. Can you state in 3–5 bullets what will change, why, and in what order?
  4. Do you know what "done" looks like and how to verify it?
  5. Which tests cover this area and will new ones be needed?

Allow-through conditions: GSD workflow or skill already invoked this session · prior edits already exist in the transcript · target file is CLAUDE.md, README.md, .gitignore, LICENSE, or any path under docs/ or .planning/.

Session flags: /tmp/bc-programmer-<session_id>.flag and /tmp/bc-planner-<session_id>.flag (suppresses bc-planner).

bc-planner PreToolUse

Fallback planning gate. Fires when bc-programmer didn't (e.g. GSD was already active at session start). Asks Claude to write a structured plan covering: what changes, why, in what order, and what risks exist — before the first file edit. Uses /tmp/bc-planner-<session_id>.flag to fire only once per session. Skips doc-only targets and sessions where prior edits are detected in the transcript.

bc-secret-guard PreToolUse

Scans file content before it is written to disk. Blocks if a real credential is found in a git-tracked file.

Detected patterns: Anthropic / OpenAI / AWS / GitHub / GitLab / Stripe / SendGrid / Slack API keys · PEM private key blocks · database connection URLs with embedded passwords · hardcoded password / token / apikey variable assignments.

Allow-through: Value matches a placeholder pattern (YOUR_KEY_HERE, process.env.X, <TOKEN>) · file is git-ignored (git check-ignore exits 0) · file is a template, example, fixture, test, or doc file.

bc-commit-msg PreToolUse

Intercepts every git commit Bash call. Extracts the -m argument and validates it:

  • Must start with a conventional commit prefix: feat fix docs chore refactor test perf ci build style revert
  • Description after the prefix must be at least 10 characters
  • Not on the vague blocklist: fix update wip misc stuff temp hack asdf

Pass-through: HEREDOC-style commits · --allow-empty · --amend --no-edit · merge commits.

bc-branch-guard PreToolUse

Protects against accidental pushes and commits to long-lived branches.

Protected branches: main · master · production · prod · release · stable

  • git push on protected branch → hard block. Suggests git checkout -b <name> && git push -u origin <name>, deriving the branch name from the last commit message.
  • git commit on protected branch → soft block. Explains the risk and asks Claude to confirm intent.
  • Per-repo opt-out: create .bc-branch-override at repo root.

bc-test-review PreToolUse

Intercepts git commit. Detects staged source files (.js .ts .py .go, etc.) vs test files. If source changed but zero test files are staged, blocks and asks Claude to review, add, or run tests first.

Quality bar enforced in the blocking prompt: assert behaviour not just execution · cover edge cases · readable test names · no padding tests.

Auto-detected runners: npm test · pytest · go test ./... · make test. Skips repos with no test infrastructure, and config/doc/lock-only commits.

bc-lint PreToolUse

Runs the project's linter and type-checker before every git commit. Skips if no code files are staged (docs/config only).

RunnerDetected by
lint-stagedpackage.json scripts
type-check / tscpackage.json scripts
lint / eslintpackage.json scripts
make lintMakefile with lint: target
go vet ./...go.mod present

Runs with a 60-second timeout via spawnSync. On failure: shows full output and blocks the commit. On pass: silently allows through.

bc-doc-sync Stop

Fires when Claude is about to finish responding. If non-doc files have uncommitted changes, blocks and requires documentation to be updated first.

Allow-through: Any file under docs/ or CLAUDE.md was updated in the same session pass.

When blocked, Claude must:

  1. Update CLAUDE.md if architecture, conventions, data flow, or key decisions changed.
  2. Update README.md if install steps, features, or usage changed.
  3. For each changed API/source file, create or update docs/<Name>.md with: Overview · How It Works · Interface · Configuration · Dependencies · Release Notes (date, summary, breaking changes).
  4. Commit and push all docs and code together in a single commit.

Release note rules: one entry per meaningful change · prepend to existing entries · breaking change = removed export, changed signature, changed config key, or changed observable behaviour.

  • /gsd:map-codebaseparallel codebase analysis agents

GSD hook scripts — gsd-check-update.js (SessionStart), gsd-context-monitor.js (PostToolUse), and gsd-statusline.js (status line) — are installed by GSD itself to ~/.claude/hooks/. They handle update checks on session start, context-window warnings as tools run, and the persistent status line display.

MCP Servers

All servers use npx — no global installs required. Node.js 18+ needed. Configured in settings.template.json under mcpServers.

Secrets & Security

The repo is safe to make public. Nothing sensitive is committed.

ItemIn repo?Notes
ANTHROPIC_API_KEY no Placeholder only in template. Fill in ~/.claude/settings.json after install.
ANTHROPIC_ADMIN_API_KEY no Optional. For org-level cost reporting. Placeholder only.
MCP server configs yes All use npx with public package names — no credentials.
Plugin identifiers yes Public marketplace names only.
Hook commands yes Template uses $HOME — no machine-specific paths.
~/.claude/settings.json no Live settings file is never tracked. Only settings.template.json is committed.
bc-secret-guard enforces this at runtime — it blocks any attempt to write a real credential to a tracked file, even accidentally during a session.
